Supplier Portal — Complete Reference

Project: Astute Bridge ERP (Audi + ASESA) Phase: Phase 1 — Week 6 Tables: 11 shared.sql tables Modules: SP-1 through SP-7 Auth: OTP → JWT RS256

1 · What the Supplier Portal Does

The Supplier Portal is a vendor-facing web interface built on top of Astute Bridge ERP's shared module layer. It gives approved vendors a controlled, read-mostly window into their own data — purchase orders, deliveries, rejections, NCRs, and performance — without granting them access to the internal ERP. The portal replaces WhatsApp and phone calls for PO acknowledgement, dispatch confirmation, and rejection responses.

Core principle: A vendor sees only rows where vendor_id = their authenticated vendor_id. No vendor can see another vendor's POs, GRNs, rejections, or prices. This is enforced at the API layer (extracted from JWT), not just by SQL WHERE clauses.

Vendor can do

  • ✅ View their active and historical POs
  • ✅ Acknowledge or decline a PO
  • ✅ Submit Advance Shipment Notice (ASN)
  • ✅ Confirm dispatch with vehicle number
  • ✅ View GRN status after delivery
  • ✅ View and acknowledge rejections
  • ✅ Submit Corrective Action Report (CAR)
  • ✅ Raise a dispute on an NCR/GRN
  • ✅ Upload delivery documents (challan, cert)
  • ✅ View their own performance scorecard

Vendor cannot do

  • ❌ Create or edit a PO
  • ❌ See another vendor's data
  • ❌ Modify inventory or stock
  • ❌ See internal production/QA details
  • ❌ See item pricing for other vendors
  • ❌ Approve/reject their own GRNs
  • ❌ See or modify the 3-way match status
  • ❌ Access any chassis, production, or BOM data
  • ❌ See other vendors' scorecard
  • ❌ Modify or delete their action history

3 · All Tables Involved

Portal-Native Tables (owned by the portal)

vendor_portal_log
One row per portal session. Stores session_token, OTP verification status, IP, device info, expiry, and invalidation timestamp.
vendor_portal_action_log
Immutable audit log. Every vendor action (PO_ACKNOWLEDGE, ASN_SUBMIT, DISPUTE_RAISE, etc.) is written here and can never be updated or deleted.
🔒 fn_block_immutable() trigger applied
asn_log
Advance Shipment Notice header. One per dispatched PO. Contains expected_date, vehicle_number, driver_name, arrival_window, status.
asn_details
ASN line items. One row per part being shipped in that ASN. Links to po_detail_id and carries qty_shipped and vendor_batch_ref.
vendor_dispute_log
Vendor-raised disputes against NCRs or GRNs. Status: OPEN → UPHELD / REVISED / CLOSED. Includes resolution note from ERP team.

Core ERP Tables the Portal Reads (vendor-scoped)

vendor_master
Source of truth for portal access. portal_enabled and portal_username control who can log in.
rm_purchase_order
Vendor sees their POs (status, total, expected date, type). Filtered by vendor_id from JWT.
rm_purchase_order_details
PO line items. Vendor sees part_number, ordered_qty, received_qty, unit_price, GST breakdown.
po_tracking_log
In-transit event log. Portal writes VENDOR_DISPATCHED event. Vendor reads their own PO tracking history.
rm_receiving_log
GRN header. Vendor sees their GRN number, received_date, vehicle_number, grn_status after delivery.
grn_detail
GRN batch lines. Vendor sees qty_received, qty_accepted, qty_rejected, batch_status per item.
rejection_log
Rejection lots. Vendor sees rejection raised against their delivery. Can acknowledge and provide response.
ncr_log
Non-Conformance Reports. Vendor reads NCR details, submits CAR (Corrective Action Report), sees resolution status.
debit_note
Financial deductions raised against vendor. Read-only on portal. Vendor sees amount, reason, status (DRAFT to SETTLED).
vendor_scorecard
Pre-computed performance metrics: on_time_pct, first_pass_pct, rejection_count, ncr_count, avg_lead_time_days.
inspection_documents
Document store. Vendor uploads test certs, delivery challans, and insurance docs here via SP-6.
rm_vendor_mapping
Vendor's approved item list. Portal reads this to show which items a vendor is approved to supply.

4 · Authentication Flow

No password login. Vendors authenticate via OTP only. Credentials = portal_username (mobile or email set by ERP admin). JWT RS256 is used — same keypair as internal ERP but different claims scope (sub_type: "vendor").

StepAPI CallTable WriteLogic
Pre-condition — vendor_master ERP Admin sets portal_enabled = TRUE and portal_username = '9876543210'
1. OTP Request POST /portal/auth/otp-request notification_queue System looks up vendor by portal_username. Generates 6-digit OTP, stores in Redis (TTL 5 min), sends via SMS/email
2. OTP Verify POST /portal/auth/otp-verify vendor_portal_log Validates OTP from Redis. On success: inserts session row with unique session_token, otp_verified=TRUE, expires_at = NOW() + 8h. Issues JWT RS256 (sub=vendor_id, sub_type=vendor, session_id=log_id)
3. Request Auth Every portal API call — JWT decoded → vendor_id extracted. vendor_portal_log checked: otp_verified=TRUE AND expires_at > NOW() AND invalidated_at IS NULL
4. Logout POST /portal/auth/logout vendor_portal_log Sets invalidated_at = NOW(). JWT is immediately invalid on next request.
5. Expiry — — Celery cron cleans expired sessions (no hard delete — audit trail kept). Vendor must re-OTP after 8 hours.
vendor_portal_log — key columns driving auth
vendor_portal_log
  session_token  VARCHAR(128) UNIQUE   — random 128-char token, stored in JWT
  otp_verified   BOOLEAN DEFAULT FALSE — flipped TRUE on /otp-verify success
  ip_address     INET                  — for security audit
  issued_at      TIMESTAMPTZ           — session start
  expires_at     TIMESTAMPTZ           — hard session expiry (8h default)
  invalidated_at TIMESTAMPTZ           — set on logout; NULL = still valid

5 · Portal Modules (SP-1 through SP-7)

SP-1
Authentication & Session Management Both Projects
Handles vendor login, OTP verification, session lifecycle, and secure logout. Gateway module — no other module works without an active SP-1 session.
Tables
vendor_master        — portal_enabled, portal_username, contact_mobile, contact_email
vendor_portal_log    — session record (one row per login attempt that passed OTP)
notification_queue   — OTP delivery (SMS via SMTP/gateway configured post-Phase 2d)
Access control check on every request
SELECT v.vendor_id, v.vendor_name, v.is_active, v.portal_enabled
FROM   vendor_master v
JOIN   vendor_portal_log s ON s.vendor_id = v.vendor_id
WHERE  s.session_token   = :token_from_jwt
  AND  s.otp_verified   = TRUE
  AND  s.expires_at     > NOW()
  AND  s.invalidated_at IS NULL
  AND  v.is_active       = TRUE
  AND  v.portal_enabled  = TRUE;
SP-2
PO View & Acknowledgement Both Projects
Vendor can see all POs raised to them. Each PO shows line items, unit prices, quantities, expected delivery date, and current status. Vendor must acknowledge within a configurable window (default 24h) or it auto-escalates. A declined PO triggers an internal ERP alert.
Tables read / written
-- READ
rm_purchase_order          — PO header (status, total_amount, expected_date, po_type)
rm_purchase_order_details  — line items (part_number, ordered_qty, unit_price, gst, line total)
po_tracking_log            — tracking events history for this PO
rm_vendor_mapping          — to display standard cost / approved item context

-- WRITE (on PO_ACKNOWLEDGE or PO_DECLINE)
vendor_portal_action_log   — action_type = 'PO_ACKNOWLEDGE' or 'PO_DECLINE'
po_tracking_log            — event_type = 'VENDOR_ACKNOWLEDGED'
rm_purchase_order          — status updated: SENT → ACKNOWLEDGED or DECLINED
Vendor-isolation query
SELECT po.*, pod.*
FROM   rm_purchase_order po
JOIN   rm_purchase_order_details pod ON pod.po_id = po.po_id
WHERE  po.vendor_id = :vendor_id_from_jwt   -- HARD ISOLATION
  AND  po.status NOT IN ('DRAFT')          -- DRAFT POs not visible to vendor
ORDER BY po.created_at DESC;
SP-3
ASN — Advance Shipment Notice Both Projects
Before the vendor dispatches goods, they must submit an ASN. This gives the ERP team advance notice of what's coming, when, and in which vehicle. The gate team uses the ASN to pre-plan receiving. The ASN auto-populates vehicle_number in rm_receiving_log when the GRN is created.
Tables written on ASN_SUBMIT
asn_log              — header: asn_number, po_id, expected_date, vehicle_number, driver_name, arrival_window
asn_details          — lines: one row per part shipped (qty_shipped, vendor_batch_ref)
vendor_portal_action_log — action_type = 'ASN_SUBMIT', entity_type='ASN', entity_id = asn_id

-- Downstream effect on rm_receiving_log when GRN created from gate:
rm_receiving_log     — vehicle_number pre-populated from asn_log.vehicle_number if asn matches PO
ASN status lifecycle
SUBMITTED  →  CONFIRMED  (ERP store team confirms expected arrival)
SUBMITTED  →  EXPIRED    (if material never arrived and window passed)
ASN line item constraint
-- qty_shipped per line must not exceed (ordered_qty - received_qty) on that PO detail
SELECT (pod.ordered_quantity - pod.received_quantity) AS remaining_qty
FROM   rm_purchase_order_details pod
WHERE  pod.po_detail_id = :po_detail_id
  AND  pod.po_id IN (
       SELECT po_id FROM rm_purchase_order WHERE vendor_id = :vendor_id_from_jwt
  ); -- isolation enforced
SP-4
Dispatch Confirmation Both Projects
After ASN is submitted and goods are loaded, vendor confirms actual dispatch. This creates the VENDOR_DISPATCHED event in po_tracking_log and increments in_transit_qty on rm_inventory — signalling to the ERP that this stock is inbound and should be counted in effective available stock for shortage calculations.
Tables written on DELIVERY_CONFIRM
po_tracking_log           — event_type = 'VENDOR_DISPATCHED', tracking_ref = vehicle/LR number
vendor_portal_action_log  — action_type = 'DELIVERY_CONFIRM'
rm_inventory              — in_transit_qty += shipped qty per part
                              -- Decremented when GRN is created and gate_log confirms arrival
asn_log                   — status: SUBMITTED → CONFIRMED
in_transit_qty lifecycle
ASN_SUBMIT         → in_transit_qty += qty_shipped   (anticipated)
DELIVERY_CONFIRM   → no change (already counted in ASN)
GRN_CREATED        → in_transit_qty -= qty_received   (goods arrived, moved to pending_putaway)
SP-5
Rejection & NCR Response Both Projects
When QA rejects a batch, the vendor is notified automatically (notification_queue). On the portal, they can see the rejection lot, acknowledge it, submit a corrective action report (CAR), and raise a dispute if they disagree. For Audi, a debit_note may also be visible. This is the most business-critical module for vendor relationship management.
Tables involved
-- READ
rejection_log             — lot-level rejection (rln, reason_code, total_qty, status, disposition_deadline)
rejection_log_details     — item-level rejection lines (part_number, qty, defect detail)
rejection_photos          — QA-uploaded photos of the rejected material (vendor can view)
ncr_log                   — NCR raised from rejection (defect_description, disposition_code, car_due)
debit_note                — financial deduction (Audi specific; read-only on portal)
vendor_dispute_log        — existing disputes raised by this vendor

-- WRITE
rejection_log             — vendor_response = 'WILL_REPLACE' / 'WILL_COLLECT' / 'DISPUTE'
                              vendor_responded_at = NOW()
ncr_log                   — status: NOTICE_SENT → ACKNOWLEDGED, then CAR_SUBMITTED
vendor_dispute_log        — INSERT on DISPUTE_RAISE (dispute_type, description)
vendor_portal_action_log  — action_type ∈ {'REJECTION_ACKNOWLEDGE', 'CAR_SUBMIT', 'DISPUTE_RAISE'}
NCR response state machine (vendor side)
NOTICE_SENT  →  ACKNOWLEDGED  (vendor reads + acknowledges NCR)
ACKNOWLEDGED →  CAR_SUBMITTED (vendor submits root cause + corrective action)
CAR_SUBMITTED → CAR_ACCEPTED  (ERP QA team accepts) or CAR_REVISION_REQUESTED
SP-6
Document Upload Both Projects
Vendors upload delivery challans, test certificates, material mill certs, and insurance documents directly to the portal. Files are stored in MinIO (Audi) or S3/Azure Blob (ASESA). Document records are linked to the GRN or PO and visible to the store team inside the main ERP.

SP-7
Scorecard & Performance Dashboard Both Projects
Read-only view of the vendor's performance metrics. The vendor_scorecard is pre-computed monthly by a Celery Beat cron. On-time delivery %, first-pass QC %, rejection count, NCR count, and average lead time are shown. The intent is transparency — good vendors should see their standing; poor performers get the data to improve.
Tables read (read-only, vendor sees only their own row)
vendor_scorecard
  vendor_id               -- isolation key
  period_start, period_end
  deliveries_total        -- total GRNs received in period
  deliveries_on_time      -- GRNs received on or before expected_date
  batches_total           -- total grn_detail batches
  batches_first_pass      -- batches with batch_status = QC_APPROVED on first inspection
  rejection_count         -- rejection_log rows in period
  ncr_count               -- ncr_log rows in period
  avg_lead_time_days      -- avg(received_date - order_date) per PO
  on_time_pct             -- GENERATED ALWAYS: deliveries_on_time / deliveries_total * 100
  first_pass_pct          -- GENERATED ALWAYS: batches_first_pass / batches_total * 100
  calculated_at

vendor_portal_action_log  — action_type = 'SCORECARD_VIEW' written on every open

Pattern: portal_auth_guard middleware (FastAPI dependency)
async def portal_auth_guard(request: Request, db: Session = Depends(get_db)):
    token = request.headers.get("Authorization").split(" ")[1]
    payload = decode_jwt_rs256(token)                    # verifies signature
    
    if payload.get("sub_type") != "vendor":              # staff JWT rejected here
        raise HTTPException(403)
    
    session = db.query(VendorPortalLog).filter(
        VendorPortalLog.session_token == payload["session_id"],
        VendorPortalLog.otp_verified  == True,
        VendorPortalLog.expires_at     > datetime.utcnow(),
        VendorPortalLog.invalidated_at is None
    ).first()
    if not session: raise HTTPException(401)
    
    request.state.vendor_id = payload["sub"]              # ALWAYS from token, never from body
    request.state.session_id = session.log_id
    return request.state
